Contraintes et éléments évalués

Le choix du domaine métier

Le choix du domaine métier est libre tant qu’il permet de respecter les contraintes du projet.

Il n’est pas nécéssaire d’avoir un domaine qui requiert du code “complexe” ou l’utilisation d’algorithmes.

Le domaine choisi doit être fourni et comprendre plusieurs entités métiers qui auront chacune leur cycle de vie ainsi que des endpoint exposés sur votre API.

Le domaine choisi doit permettre l’inclusion d’au moins un traitement asynchrone (Il n’y a pas besoin que ce soit très complexe, il peut simplement s’agir d’un traitement de fichier spécifique à votre domaine)

1. L’architecture globale attendue

Une interface utilisateur

Votre application doit être accessible via une interface utilisateur. Le choix de la technologie est libre. Il peut s’agir d’une application web, d’une application mobile ou bien d’un client lourd.

Un backend

Votre application doit contenir au moins un service de type “Backend” qui va être responsable de la persistance de vos données et de la coordination des opérations métier. Votre backend devra exposer une API (REST de préférence, mais il est possible de faire du gRPC par exemple si vous le souhaitez)

Le backend devra être appelé uniquement par votre frontend.

Votre Backend ne doit en aucun cas faire confiance à votre frontend. Vous devrez valider les tokens reçus à chaque appel (ou mettre en place un middleware qui se charge de cette partie)

La technologie est libre là aussi mais il est imposé d’avoir des annotations de types et une évaluation statique de votre typage. Choisissez donc un langage qui supporte ces contraintes.

Quelques précisions :

  • Le JS même si vous utilisez de la JSDoc pour documenter vos fonctions n’est pas accepté. Préférez le Typescript.
  • Si vous utilisez du Typescript, il est recommandé d’éviter le type “any” (sauf en cas de force majeure ou pour du mock lors de tests unitaires). Préférez l’utilisation du type “unknown” avec du rétrécissement lorsque c’est applicable.
  • Python permet l’utilisation d’annotations de type optionnelles et l’ajout de MyPy peut les rendre obligatoire. Cette combinaison est acceptée pour ce projet si vous souhaitez utiliser du Python.

Un service d’IAM (Identity & Access management)

Votre architecture devra inclure un système d’IAM moderne. Il est conseillé de mettre en place un système robuste existant tel que Keycloak, Firebase, Cognito ou autre plutôt que de le développer vous même.

Ce système sera atteignable par le frontend pour gérer la connexion à votre application et l’obtention de tokens pour prouver votre identité.

Votre backend pourra atteindre ce service pour faire des appels à leur API d’administration pour vérifier la validité d’un token, envoyer des mails, modifier des données utilisateurs ou autre.

Votre système d’IAM doit à minima gérer via ses propres écrans :

  • La création d’un compte
  • La connexion
  • Le reset du mot de passe
  • La mise en place d’une authentification forte (Je vous recommande d’utiliser l’application Aegis pour gérer les codes 2FA sur votre téléphone)
  • La suppression de l’authentification forte

Le design des écran ne sera PAS évalué. Vous pouvez utiliser les écrans par défaut même si le thème ne correspond pas au design de votre interface utilisateur.

Un service de type Worker

Votre application devra contenir au moins un service de type Worker, qui effectuera des tâches asynchrones telles que l’envoie de mail / SMS, l’upload de fichier ou le traitement d’image, ainsi que toutes tâches liées à votre domaine métier.

Votre Worker doit inclure au moins une tâche spécifique à votre domaine (Génération de PDF, validation et analyse d’un fichier upload par un utilisateur)

Votre Worker et votre Backend devront communiquer par un système de Queue (SQS, Pub/Sub, Kafka, RabbitMQ). Vous devrez mettre en place deux queues. Une pour gérer les demandes (Backend -> Worker) et une pour les réponses (Worker -> Backend). Le Worker ne doit PAS appeler de manière synchrone votre Backend.

Votre Worker peut gérer plusieurs tâches différentes, il n’est pas nécéssaire d’avoir un worker par tâche. Il est possible d’en rajouter si vous le souhaitez, mais ce n’est pas obligatoire.

Votre Worker devra être complètement isolé de votre Backend. Il ne doit pas avoir accès à la base de données que gère votre Backend (Si votre besoin le justifie, avoir un accès en lecture seule au maximum) ni aux modèles existants. Il doit recevoir les informations dont il a besoin pour son traitement en entrée et fournir une réponse (Même en cas d’échec) au Backend via une autre Queue. Votre Backend est responsable de la gestion de la base de données.

Votre Worker sera responsable de la politique de retry. Pour chaque tâche, vous devrez déterminer la criticité et la bonne politique de retry à appliquer.

Vous aurez des points en plus si le Worker est développé en Rust.

2. L’architecture logicielle

L’architecture pour votre projet n’est pas imposée, mais elle sera évaluée selon certaines contraintes que vous devrez respecter au maximum.

Utilisation d’ADR (Architecture Decision Record)

Il est attendu que vous rédigiez des ADR qui devront justifier vos décisions d’architecture logicielle (entre autre). Vos ADR doivent être datés et versionnés dans les différents repository git de vos projets. Le template est libre mais je vous recommande un template simplifié qui ne vous demandera pas trop de rédaction. Le but est qu’il soit facile de rédiger un ADR sinon personne n’adoptera ce système dans votre équipe.

Voici une liste des situations dans lesquelles un ADR est attendu :

  • Un changement dans un modèle
  • L’ajout d’une dépendance externe
  • Un choix d’architecture

Vous êtes libre d’ajouter des ADR dans d’autres situations si vous le souhaitez.

Test unitaires

Il est attendu que votre code ait des tests unitaires, surtout sur les parties critiques comme votre domaine métier. Vos tests unitaires seront un gage de qualité de votre architecture. Vous devez penser votre code pour qu’il soit facilement testable.

Isolation des dépendances externes

Vos projets devront suivre une architecture capable d’isoler au maximum vos dépendances externes (Même dans votre interface utilisateur). Vous devrez faire usage d’une architecture qui isolera votre domaine au maximum de toutes dépendances externes. Vous avez le droit d’utiliser des dépendances externes dans votre domaine si vous le justifiez par un ADR. Quelques exemples courants en TypeScript peuvent être Neverthrow ou Immutable qui sont des dépendances avec des interfaces très stables et aucunes peer-dependencies.

Gestion d’erreur

Vos projets devront mettre en place une gestion d’erreur appropriée. Pour votre backend, vous devrez séparer les erreurs métier des erreurs techniques. Ne renvoyez pas de code d’erreur HTTP dans votre métier par exemple.

Observabilité

Votre Backend et votre Worker doivent produire des logs qui permettent de suivre l’exécution du code et de vos traitement asynchrones de bout en bout. Pour cela, vous pourrez utiliser des id de corrélation. Vous pouvez aller plus loin et mettre en place un système de traces et de métriques mais ce n’est pas obligatoire (OpenTelemetry + Prometheus + Grafana par exemple)

3. Le DevOps

Utilisation d’un gestionnaire de version

Votre projet devra utiliser un logiciel de gestion de versions tel que Git. Vous devrez mettre en place un flow de développement documenté que vous devrez respecter (gitflow, trunk-based, gitlab-flow etc).

Présence d’une pipeline d’intégration et de livraison continue

Vous devrez mettre en place une pipeline d’intégration et de livraison continue. Cette pipeline devra complémenter votre workflow git et devra donc être implémentée tôt dans votre projet.(du moins une première version basique) Voici les attentes sur vos différentes pipelines :

  • Avoir une pipeline automatique qui se lance à chaque commit de la branche main
  • Cette pipeline doit avoir au minimum les étapes suivantes :
    • Des test unitaires
    • Une vérification par un linter
    • Un build du projet
    • Une mise en ligne du livrable (Sur Docker Hub par exemple)

Vous êtes libres de rajouter d’autres étapes ou pipelines selon vos besoin.

Présence d’une pipeline de déploiement en production

Le déploiement sur votre infrastructure cloud sera considéré comme une mise en production. Il est attendu que vous ayez à disposition une pipeline manuelle ou automatique qui permette de mettre à jour un projet sur votre environnement cloud. Le déploiement des ressources IAC n’est pas obligatoirement inclus dans cette étape.

Utilisation de conteneurs

Vos livrables seront sous la forme de conteneurs. Vous pouvez utiliser https://hub.docker.com/ pour mettre en ligne vos livrables ou bien des registres privés tels que les registres privés Gitlab. Vos conteneurs doivent impérativement être mis en ligne sur un registre. La qualité de votre Dockerfile sera aussi évaluée selon certain critères:

  • Optimisation de la taille de l’image
  • Fiabilité de l’image (Qui en est l’auteur, est-elle maintenue)
  • Capacité à justifier le choix du tag de l’image
  • Optimisation des étapes pour maximiser le cache Docker
  • Utilisation d’un build de production au sein de l’image (pas de npm run dev par exemple)

Déploiement sur un Cloud

Votre projet devra être déployé sur un cloud et accessible via HTTPS sur un nom de domaine. Je suis en mesure de vous fournir des sous domaines si besoin pour vos projets pour vous éviter d’en acheter un. Vous pouvez utiliser Kubernetes mais je ne le recommande pas car cela vous entrainerai des coûts. Je vous conseille d’utiliser les ressources gratuites sur Google Cloud (Cloud Run et Pub/Sub) ou bien AWS (SQS + Lambda).

Utilisation de l’IAC

Vos ressources Cloud devront être déployées automatiquement via des scripts d’IAC. Dans l’idéal, vous devrez avoir un repository Git contenant toutes vos ressources Cloud. Vos scripts IAC devront être en mesure de supporter plusieurs environnements (dev / Prod) même s’il n’est pas demandé de déployer plusieurs environnement dans ce projet pour vous éviter au maximum d’engendrer des coûts.

4. Gestion des breaking changes

Votre projet doit être en mesure de gérer le déploiement de breaking changes sans impacter votre interface utilisateur. Vous devez considérer que votre frontend et votre backend sont développés par deux équipes différentes qui n’auront pas forcément le même rythme ni les mêmes priorités. Votre Backend doit être en mesure de déployer une breaking change sans que cela n’impacte le frontend. Voici quelques exemples d’options pour gérer ces cas :

  • Mettre en place un système de versionning pour votre API
  • Mettre en place un système de versionning sur vos schémas d’entrées
  • Mettre en place un système de feature flags qui vous permettra d’activer une breaking change une fois que le frontend est à jour

Vous devrez mettre en place une procédure et la documenter dans un fichier qui sera versionné avec le reste de votre projet.

Pour le rendu final, votre backend devra contenir au moins une route contenant une breaking change. Vous devrez être en mesure de démontrer en direct que votre frontend peut utiliser l’une ou l’autre des versions selon votre système. Si vous avez choisi de versionner votre schéma d’API par exemple, vous pourrez déployer une version du frontend utilisant la première version du schéma, puis, déployer une deuxième version plus récente qui utilisera la nouvelle. La seule contrainte est que votre backend ne devra PAS être redéployé.
